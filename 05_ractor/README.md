# 05. Ractor - Rubyの並列実行モデル

## 概要

Ractor（Ruby Actor）はRuby 3.0で導入された並列実行のための機構です。従来のThreadモデルではGVL（Global VM
Lock）により真の並列実行が制限されていましたが、RactorはこのRuby制約を打破し、マルチコアCPUの性能を最大限に活用できます。

## GIL/GVLの制限とRactorの必要性

### GVL（Global VM Lock）とは

Ruby（CRuby/MRI）にはGVL（旧称GIL: Global Interpreter
Lock）が存在します。これは同時に1つのスレッドしかRubyコードを実行できないようにするロック機構です。

```text

┌─────────────────────────────────────────────┐
│              Rubyプロセス                     │
│                                              │
│  ┌──────┐  ┌──────┐  ┌──────┐               │
│  │Thread│  │Thread│  │Thread│   ← 複数存在   │
│  │  1   │  │  2   │  │  3   │               │
│  └──┬───┘  └──┬───┘  └──┬───┘               │
│     │         │         │                    │
│     └────┬────┘─────────┘                    │
│          ▼                                   │
│    ┌──────────┐                              │
│    │   GVL    │  ← 同時に1つのThreadのみ通過  │
│    └──────────┘                              │
│                                              │
└─────────────────────────────────────────────┘

```

### GVLの影響

| 処理タイプ | GVLの影響 | Thread並列化の効果
| ----------- | ---------- | ------------------
| CPU-bound（計算処理） | 常にGVLを保持 | 効果なし（逐次実行と同等）
| I/O-bound（ネットワーク・ファイル） | I/O待ち中はGVL解放 | 効果あり
| C拡張（ネイティブコード） | 実装依存でGVL解放可 | 実装による

### Ractorが解決する問題

Ractorは各インスタンスが独自のGVLを持つため、複数のRactorが真に並列に実行できます。

```text

┌─────────────────────────────────────────────────┐
│                Rubyプロセス                       │
│                                                  │
│  ┌────────────────┐  ┌────────────────┐          │
│  │   Ractor 1     │  │   Ractor 2     │          │
│  │  ┌──────────┐  │  │  ┌──────────┐  │          │
│  │  │  GVL 1   │  │  │  │  GVL 2   │  │  ← 独立 │
│  │  └──────────┘  │  │  └──────────┘  │          │
│  │  独自のメモリ空間 │  │  独自のメモリ空間 │          │
│  └────────────────┘  └────────────────┘          │
│          ↕ メッセージパッシング ↕                   │
└─────────────────────────────────────────────────┘

```

## Ractorが有効なケース

### 適しているケース

1. CPU-bound処理の並列化

   - 大量の数値計算（暗号処理、画像処理、データ変換）
   - 素数判定、フィボナッチ計算などの数学的処理
   - JSON/XMLの大量パース処理

2. Fan-out/Fan-inパターン

   - 独立した計算タスクを複数ワーカーに分散します
   - MapReduceスタイルのデータ処理に使います

3. パイプライン処理

   - データ変換の各ステージを独立したRactorとして構築します
   - ストリーム処理パターンに使います

### 適さないケース

1. I/O-bound処理にはThreadで十分です
2. 共有状態が多い処理はRactorの分離ルールと相性が悪いです
3. 既存ライブラリに依存する処理は多くのgemがRactor非対応です

## Thread / Ractor / Processの比較

| 特性 | Thread | Ractor | Process
| ------ | -------- | -------- | ---------
| メモリ空間 | 共有 | 分離（メッセージパッシング） | 完全分離
| GVL | 共有（1つ） | 各Ractor独立 | 各Process独立
| CPU並列実行 | 不可（CRuby） | 可能 | 可能
| I/O並列実行 | 可能 | 可能 | 可能
| 通信コスト | 低（共有メモリ） | 中（コピー/移動） | 高（IPC）
| 生成コスト | 低 | 中 | 高
| データ競合リスク | 高（要Mutex） | なし（設計で排除） | なし
| デバッグ容易性 | 難（競合状態） | 中（分離保証） | 易（独立性）
| ライブラリ互換性 | 高 | 低（experimental） | 高
| 適用シーン | I/O多重化 | CPU-bound並列化 | プロセス分離

## 通信パターン

### Push型（send/receive）

外部からRactorにデータを送り込むパターンです。プロデューサー・コンシューマーモデルに適しています。

```ruby

ractor = Ractor.new do
  msg = Ractor.receive  # メッセージを待ち受ける
  "処理結果: #{msg}"
end

ractor.send("データ")   # メッセージを送信
result = ractor.take     # 結果を取得

```

### Pull型（yield/take）

Ractorが結果を公開し、外部から取り出すパターンです。ジェネレーターパターンに適しています。

```ruby

ractor = Ractor.new do
  Ractor.yield("結果1")
  Ractor.yield("結果2")
end

first = ractor.take   # "結果1"
second = ractor.take  # "結果2"

```

## 分離ルールと共有可能オブジェクト

### 共有可能（Shareable）なオブジェクト

- 数値（Integer, Float, Rational, Complex）
- Symbol
- true, false, nil
- Frozen文字列
- Frozen配列・ハッシュ（要素もすべてshareable）
- Ractorオブジェクト自体
- `Ractor.make_shareable`で変換されたオブジェクト

### 通信時のセマンティクス

| 方式 | 動作 | 用途
| ------ | ------ | ------
| コピー（デフォルト） | ディープコピーを作成 | 安全だがメモリ・速度コストがあります
| 移動（`move: true`） | 所有権を移転し、元はアクセス不可になります | 大きなオブジェクトの効率的な受け渡しに使います
| 共有（shareableオブジェクト） | 参照を共有します | immutableデータの効率的な共有に使います

## プロダクション利用の現状（2025年時点）

### ステータス

RactorはRuby 3.0で導入されて以降、experimental（実験的）の位置付けが続いています。Ruby
3.3/3.4時点でも`Warning[:experimental]`が発生します。

### プロダクション利用の課題

1. gem互換性の不足

   - 多くのgemがRactorに対応していません
   - グローバル変数やクラス変数を使用するgemはRactor内で動作しません
   - ActiveRecord等のRailsコンポーネントはRactor非対応です

2. APIの安定性

   - 今後のバージョンでAPI変更の可能性があります
   - エラーメッセージやエラークラスが変更される可能性があります

3. デバッグツールの不足

   - Ractor対応のデバッガやプロファイラがまだ発展途上です
   - スタックトレースの解読が困難な場合があります

4. パフォーマンス特性

   - オブジェクトのコピーコストが無視できない場合があります
   - Ractor生成コストがThreadより高いです
   - 小さなタスクでは並列化のオーバーヘッドが利点を上回ります

### 推奨される利用方針

- 学習・実験目的: 積極的に活用し、並列プログラミングの概念を理解します
- 社内ツール・バッチ処理: 限定的な範囲で試験的に導入できます
- 本番Webアプリケーション: 現時点では慎重に判断してください。Railsとの統合はまだ現実的ではありません
- CPU-bound特化のマイクロサービス: 最も適した利用シーンです
